# 稳定期问题修复边界守则

## 1. 文档定位

本文件用于解决一个“架构进入稳定期后”的协作问题：

- 你会持续发现具体问题
- 其中很多问题本身是小问题，看起来可以快速处理
- 但如果每次都直接让 Codex 写代码，代码可能被放进不属于该问题的层
- 久而久之，会重新出现职责回流、边界变脏、功能分散

本文件的目的不是增加流程负担，而是提供一个简单可复用的分流方案：

- 什么问题可以用小修执行单处理
- 什么问题要先做边界判断
- 什么问题必须先转成执行单

如果与 [`architecture-target.md`](./architecture-target.md) 冲突，以长期架构方向为准。

---

## 2. 适用范围

本文件主要服务于：

- 用户
- 主规划 / 边界判断 / 验收助手

它主要适用于：

- 稳定期的小 bug、小回归、小行为修正
- 你担心“这题可能写错层”的中等风险问题
- 明显涉及迁职责、跨层、结构债务暴露的问题

它不直接替代：

- 长期架构目标文档
- UI 长期规范文档
- 一次性执行单

它做的是：

- 先判断当前问题属于哪一种协作模式
- 再决定是否进入实现

---

## 3. 要解决的核心风险

稳定期最常见的不是“大重构做坏”，而是“小修复慢慢把边界重新打散”。

典型风险包括：

- 为了图快，把逻辑补进当前最容易改的文件，而不是正确归属层
- 为了修一个点，顺手新增 page-local helper、万能 util、临时 facade
- 让页面重新直接碰基础设施
- 让 `app` 层重新承接数据细节
- 让 `commands/*` 或根层遗留模块重新变厚
- 把本来属于一个 feature 的修复，拆散到多个不稳定位置

所以稳定期的重点不是把小问题变成重流程，而是“先判断这是谁的问题，再用足够轻的小修执行单交给正确执行主体”。

---

## 4. 核心方法：先分流，再执行

以后遇到问题，默认先把它分到下面 3 种模式之一。

但要注意：

- 这里的“小修执行单模式 / 边界判断模式 / 执行单模式”首先是在说问题类型与边界复杂度
- 它不自动等于“由主助手亲自改代码”
- 当前仓库的默认协作约定仍是：主助手先分流、定边界、写小修执行单或一次性执行单，再交给 GPT-5.3-Codex / 中 执行；只有用户明确说“你来修”或等价表达时，主助手才直接动手改代码

### 模式 A：小修执行单模式

适用条件：

- 问题归属层清楚
- 当前已有明确 owner 文件或 owner 模块
- 修复可以留在同一层或同一 feature 内完成
- 不需要新建跨层抽象

执行要求：

- 主助手先给出小修执行单，至少明确 owner、允许修改范围、非目标和验收点
- 默认交给 GPT-5.3-Codex / 中 按小修执行单实现
- 执行方只做最小修复
- 不新增新的跨 feature helper
- 不把代码迁到更“方便改”的层
- 不顺手做额外架构整理

### 模式 B：边界判断模式

适用条件：

- 问题本身不大，但归属层不完全清楚
- 可能有两个以上合理落点
- 一旦落错层，就容易引发职责扩散

执行要求：

- 先判断“当前 owner 应该是谁”
- 先明确允许修改的层和禁止扩散的层
- 判断完成后再形成小修执行单或一次性执行单

### 模式 C：执行单模式

适用条件：

- 修复必然跨层
- 修复过程中需要迁移职责
- 当前问题其实暴露的是一段结构性债务
- 如果不先定范围，Codex 很容易一边修 bug 一边做隐性重构

执行要求：

- 先产出一次性执行单
- 锁定范围、非目标、验收门槛
- 按阶段执行并验收

---

## 5. 默认判断顺序

以后发现一个问题时，按下面顺序判断：

1. 这是不是单点行为 bug，而不是结构问题。
2. 这个问题当前的 owner 层是否明确。
3. 最小修复能不能留在 owner 层内部。
4. 如果不能，是否已经进入“边界判断”或“执行单”范围。

可以把它简化成一句话：

先问“这是谁的问题”，再问“怎么修”。

---

## 6. 一个实用判断规则

如果你希望流程足够轻，可以长期用下面这条规则：

- 能在“现有 owner 模块内”修完的，先出小修执行单，再交给 GPT-5.3-Codex / 中 实现
- 需要“跨 owner 模块协作”但不确定怎么落点的，先做边界判断
- 需要“新增抽象 / 迁移职责 / 改依赖方向”的，先出执行单

这条规则比“按文件多少判断”更有效。

因为真正危险的不是改了几个文件，而是：

- 有没有跨层新增职责
- 有没有把代码放进错误落点
- 有没有让未来问题更难收口

---

## 7. 给 Codex 的默认约束

如果你以后想继续直接问问题，可以给 Codex 一个长期默认约束。

### 7.1 对小修执行单模式

- 先判断当前问题的 owner 层和 owner 模块
- 先输出小修执行单，不默认直接写实现代码
- 只允许在 owner 边界内做最小修复
- 不新增跨层 helper、facade、service、repository
- 如果最小修复会迫使代码跨层扩散，停止编码，改为输出边界风险说明

### 7.2 对边界判断模式

- 先输出“建议归属层、允许写入范围、禁止扩散范围”
- 再决定是生成小修执行单，还是升级为一次性执行单
- 如果需要跨层迁移，自动升级为执行单模式

### 7.3 对执行单模式

- 不先写代码
- 先写一次性执行单
- 明确目标、范围、非目标、验收方式

---

## 8. 默认执行主体约定

为避免稳定期问题再次退回“看到问题就直接写代码”的模式，当前默认执行主体约定如下：

- 默认由主助手负责：问题分流、owner 判断、边界约束、执行单、验收
- 默认由 GPT-5.3-Codex / 中 负责：按小修执行单或一次性执行单实现

其中“中”表示实现阶段默认使用中等推理强度。
- 只有当用户明确说“你来修”“这次你直接改”或等价表达时，主助手才直接修改代码

也就是说：

- 主助手默认不直接写实现代码，小问题也不例外
- owner 明确的小问题默认进入“小修执行单模式”，由主助手先写清小修执行单
- 实施主体默认仍然是 GPT-5.3-Codex / 中，除非用户明确切换

---

## 9. 什么时候必须先停下来

出现下面任一情况，就不要继续按“小修执行单模式”推进，而应先停下来：

- 修复需要新建 `shared/*` 能力才能落地
- 修复会让页面重新直接依赖基础设施
- 修复会把数据访问逻辑放回 `app/*`、`commands/*` 或前端页面层
- 修复会让 `src/lib/*` 再次增长成临时万能层
- 修复看起来像 bug，但本质是边界缺失
- 你自己也说不清“这段代码到底该放哪里”

这些信号一出现，继续按小修处理通常比先停下来更贵。

## 10. 推荐工作流

我建议以后默认采用下面这个工作流。

### 工作流 1：日常小问题

适用于明确归属的小 bug、小回归、小文案、小空值处理、小状态同步问题。

流程：

1. 直接提问题
2. 先按“小修执行单模式”判断 owner 和最小落点
3. 默认先给一份小修执行单，交给 GPT-5.3-Codex / 中 执行
4. 修完后验收时重点看它有没有越界

如果你希望主助手亲自处理，应明确补一句：

- `这次你来修`
- `不要给 Codex，这次你直接改`
- 或其他等价表达

### 工作流 2：中等风险问题

适用于“问题不大，但你担心它会把代码写错层”的情况。

流程：

1. 先让 Codex 判断 owner 层与落点
2. 由主助手先输出边界判断或执行单
3. 只要落点明确，再交给 GPT-5.3-Codex / 中 实现
4. 如果判断中出现跨层迁移，再升级成执行单

### 工作流 3：结构性问题

适用于一眼就看出来会跨层、会迁职责、会影响多个边界的情况。

流程：

1. 先由主助手出执行单
2. 再让 GPT-5.3-Codex / 中 按执行单落地
3. 最后按执行单验收

## 11. 可直接复用的提问模板

下面这几句可以直接长期复用。

### 模板 A：小修执行单模式

```text
请按“小修执行单模式”处理这个问题：先判断当前问题的 owner 层和 owner 模块，输出小修执行单，明确允许修改范围、非目标和验收点；实现阶段默认交给 GPT-5.3-Codex / 中，只允许在现有 owner 边界内做最小修复，不新增跨层 helper / facade / service / repository，不顺手做额外重构；如果最小修复会迫使代码跨层扩散，就先停下来说明边界风险，不要直接写代码。
```

### 模板 B：边界判断模式

```text
这个问题我担心会写到错误层。请先不要改代码，先判断它的 owner 层、推荐落点、允许修改范围、禁止扩散范围；如果确认可以在正确边界内做最小修复，就输出小修执行单并交给 GPT-5.3-Codex / 中 实现；如果需要跨层迁移，就改为先出执行单。
```

### 模板 C：执行单模式

```text
这个问题不要直接修。请先按现有架构方向给出一份一次性执行单，明确目标、范围、非目标、阶段、验收门槛；等执行单确认后再进入实现。
```

## 12. 验收时重点看什么

以后验收一个“小修执行单”问题时，不只看它有没有修好，还要看这 5 件事：

1. 代码是不是还留在原本应属的层里。
2. 有没有为了修一个点新增新的跨层抽象。
3. 有没有让页面、`app`、`commands`、`src/lib/*` 重新回胖。
4. 有没有把一个 feature 的逻辑打散到不稳定位置。
5. 如果这次其实触发了结构问题，执行方有没有主动停下来升级流程。

如果功能修好了，但这 5 条里出现明显越界，就不应算“稳定期的合格修复”。

---

## 13. 最推荐的落地方式

如果你想把成本降到最低，建议以后默认这样用：

- 普通问题：直接发问题，由主助手判断后默认给小修执行单，再交给 GPT-5.3-Codex / 中
- 你自己已经感觉“这题可能会写错层”：直接发“模板 B”
- 明显涉及迁职责或跨层：直接让我先写执行单
- 只有你明确说“你来修”时，主助手才直接改代码

这样你不需要每次都重新设计流程，但也不会回到“什么都直接修，最后边界又脏掉”的状态。

## 14. 换对话框后的恢复方式

新对话不会天然记住旧对话里的协作约定。

所以这套方案的正确承载方式不是“记忆”，而是：

- 由仓库内长期文档保存规则
- 由新对话启动口令恢复当前协作模式

也就是说，换对话框以后，新的主助手应该：

1. 先读本文件
2. 再按本文件对问题进行分流
3. 再决定是输出小修执行单、先做边界判断，还是先写一次性执行单

如果没有这一步，新对话就很容易退回“看到问题就直接写代码”的默认行为。

## 15. 默认新对话启动口令

下面这句是以后最值得长期复用的默认启动口令。

它的用途是告诉新的主助手：

- 当前仓库已进入稳定期
- 默认先按本文件分流问题
- 由主助手负责边界判断、执行单和验收
- 默认由 GPT-5.3-Codex / 中 负责按执行单实现
- 只有在用户明确要求“你来修”时，主助手才直接改代码

可直接复用：

```text
本仓库现在按稳定期流程协作。请先阅读 docs/issue-fix-boundary-guardrails.md，并按其中规则判断当前问题属于小修执行单模式、边界判断模式还是执行单模式。默认由你负责 owner/边界判断、执行单和验收；实现阶段默认交给 GPT-5.3-Codex / 中 按小修执行单或一次性执行单执行。只有当我明确说“你来修”或等价表达时，你才直接改代码。即使这是 owner 明确且不跨层的最小修复，也先输出小修执行单，不要直接开始写代码。
```

## 16. 作用对象说明

本文件主要服务于：

- 用户
- 主规划 / 边界判断 / 验收助手

它不是给执行模型单独决定“我该不该直接重构”的自由授权。

执行模型在实现阶段应优先遵循：

- 已确认的小修执行单或一次性执行单
- [`AGENTS.md`](../AGENTS.md)
- [`architecture-target.md`](./architecture-target.md)

之所以不把“主助手负责分流、默认由 GPT-5.3-Codex / 中 落地、主助手负责验收，且只有用户明确说你来修时主助手才直接改代码”这套协作分工直接写进 `AGENTS.md`，是因为 `AGENTS.md` 会被执行模型一起读取，容易把角色边界写乱。
