# 产品原则与范围

## 1. 文档定位

本文件定义 `Time Tracker` 作为个人开源产品的长期产品原则、目标范围与非目标范围。

它不是某一轮需求整理记录，也不是某次版本的功能清单，而是以后判断“这个功能该不该做、该做到什么程度、现在先不做什么”的长期产品基线。

本文件主要解决 4 件事：

- 解释这个产品到底在解决什么问题
- 说明当前产品应优先服务什么用户
- 明确什么能力属于当前长期范围
- 明确什么能力暂时不属于当前产品方向

如果某次功能讨论与本文件冲突，应先回到产品原则做判断，而不是默认“用户提了就做”。

## 3. 产品一句话定义

`Time Tracker` 是一个面向个人桌面工作的、本地优先、自动记录、可信且可回看的时间追踪工具。

这句话里的每个词都很重要：

- `个人`：当前优先服务单个使用者，而不是团队协作场景
- `桌面工作`：优先围绕桌面端真实工作流，不按移动端习惯设计
- `本地优先`：默认假设数据保存在本地即可成立
- `自动记录`：产品价值建立在减少手动打点，而不是要求用户坚持手工输入
- `可信`：记录结果、边界处理、数据控制都必须让人放心
- `可回看`：不只是采集，还要能理解、校正和使用这些记录

---

## 4. 当前核心用户

当前长期优先服务的是以下用户：

- 在 Windows 桌面上长时间工作的人
- 希望低打扰地了解自己时间分布的人
- 不想依赖云端账号或订阅才能使用核心功能的人
- 希望保留本地控制权、备份权和清理权的人

当前不优先服务的用户包括：

- 需要多人协作、团队报工、组织级权限控制的用户
- 以移动端为主要使用场景的用户
- 把产品当作强激励游戏、番茄钟社交工具或绩效系统的人
- 需要完整 SaaS 报表平台、审批流或企业计费系统的用户

---

## 5. 产品要解决的核心问题

长期上，`Time Tracker` 主要解决下面几类问题：

### 5.1 自动记录问题

用户不想靠手动开始/停止计时器来保持记录完整性。

所以产品应优先提供：

- 自动识别当前前台应用
- 相对稳定的会话切分
- 对 AFK、锁屏、休眠等边界的正确处理

### 5.2 可相信的问题

时间追踪工具一旦“看起来会记错”，用户就会很快放弃。

所以产品应优先提供：

- 可解释的统计逻辑
- 尽量少漏记和少串记的边界处理
- 本地优先的数据存储与恢复能力
- 清楚的用户控制权

### 5.3 可读可管的问题

原始采样并不等于可用产品，用户还需要看懂并管理它。

所以产品应优先提供：

- 可读的 Dashboard 和 History
- 可管理的应用分类与颜色系统
- 明确的排除、改名、归类、标题记录控制

### 5.4 长期陪伴的问题

这个产品不是一时兴起的炫技工具，而是长期陪伴型桌面工具。

所以产品应优先提供：

- 安静稳定的体验
- 足够可信的数据边界
- 长时间使用不累的 UI
- 可渐进演进的结构与版本体系

---

## 6. 长期产品原则

### 6.1 可信优先于功能堆叠

如果一个新功能会削弱用户对记录准确性、数据边界或产品稳定性的信任，它就不应该优先于核心追踪正确性。

### 6.2 自动优先于手工流程

本产品的主要价值来自“自动记录 + 后续修正”，而不是让用户自己维护复杂的手工时间系统。

允许有手动校正能力，但不应把产品主路径变成手工记账器。

### 6.3 本地控制优先于外部依赖

长期默认假设：

- 本地存储应该足够成立
- 备份/恢复是核心能力
- 不应把账号体系、在线依赖、云端配置当作默认前提

### 6.4 清晰优先于炫技

不论是产品功能还是 UI 呈现，都应优先帮助用户理解自己的时间，而不是增加表演性体验。

### 6.5 渐进优先于铺得太大

个人开源产品最容易失败的方式之一，是过早把目标扩成“大而全平台”。

因此长期原则是：

- 先把核心路径做可信
- 再把高频体验做顺
- 最后才考虑大范围扩展

---

## 7. 当前长期范围

当前属于长期主方向的能力包括：

### 7.1 可信的自动追踪内核

- 前台应用识别
- 会话切分
- AFK / 锁屏 / 休眠边界处理
- 崩溃恢复与异常封口

### 7.2 可读的时间回看能力

- Dashboard 汇总
- History 视图
- 聚合统计与时间线解释一致性

### 7.3 可管理的应用语义层

- 应用分类
- 重命名
- 配色
- 排除统计
- 标题记录开关

### 7.4 本地数据控制

- 本地数据库
- 导出备份
- 恢复备份
- 清理历史记录

### 7.5 安静专业的桌面体验

- Quiet Pro 基线下的核心页面体验
- 最小但稳定的设置能力
- Windows 桌面行为一致性

---

## 8. 当前明确非目标

下面这些方向当前不属于长期主方向，除非产品阶段发生明确变化：

### 8.1 团队协作与 SaaS 化

暂不以这些为目标：

- 团队工作区
- 多人共享报表
- 管理员权限体系
- 在线组织与成员管理
- 云端订阅计费

### 8.2 重型生产力平台化

暂不以这些为目标：

- 项目管理平台
- 任务系统
- 完整 OKR / KPI / 报工体系
- 企业级运营后台

### 8.3 娱乐化效率产品路线

暂不以这些为目标：

- 强游戏化激励
- 连续签到、成就墙、社交排行榜
- 以刺激感替代可信度的交互设计

### 8.4 过早的多平台扩张

当前不应把主要资源优先投入到：

- 手机优先体验
- Web SaaS 首发
- macOS / Linux 的同步铺开

这不代表永远不做，而是当前阶段不应让它抢走核心桌面产品的注意力。

### 8.5 与核心价值弱相关的花式洞察

例如：

- 复杂 AI 总结
- 大量装饰性可视化
- 不是高频核心需求的“高级分析实验”

只有当核心记录、信任和回看已经足够稳时，这类方向才值得进入更高优先级。

---

## 9. 新功能准入标准

一个新功能要进入长期主方向，至少应同时满足以下大部分条件：

- 它能增强自动记录、可信度、可回看性或本地控制力中的至少一项
- 它服务的是当前核心用户，而不是假想中的另一类产品用户
- 它不会明显削弱现有核心路径的简单性与可信感
- 它不要求仓库立刻承担完全不同的产品形态
- 它的长期维护成本与产品收益大体匹配

如果一个提案更像是：

- 看起来很酷
- 适合截图展示
- 只服务低频边缘场景
- 会把产品引向另一类工具

那它默认不应进入主方向。

---

## 10. 砍需求或延后需求的判断口径

以下情况通常应当选择延后，而不是立即实现：

- 与当前核心价值关系弱
- 需要很高长期维护成本
- 会显著拉高 UI / 架构复杂度
- 主要为未来假想规模服务，而不是当前真实使用
- 会把产品从“个人桌面工具”拉向“平台化产品”

以下情况通常应当考虑直接否掉：

- 会破坏本地优先立场
- 会让数据边界更难解释
- 会把产品重心转成团队 SaaS、游戏化或营销型工具
- 会要求大规模新表面，但没有稳定用户价值支撑

---

## 11. 当前阶段最重要的产品判断

在当前阶段，最重要的不是“功能面更大”，而是：

- 让追踪结果更可信
- 让数据控制更稳
- 让四个核心页面更清楚
- 让产品边界更自洽
- 让用户越来越能把它当成长期桌面工具

所以如果两个方向同时竞争资源，默认优先级应是：

1. 可信记录与数据安全
2. 核心回看与管理体验
3. 结构稳定与可持续维护
4. 新扩展功能

---

## 12. 给 Codex 与后续协作者的默认约束

当仓库内出现新想法、新 feature 或产品扩展提案时，默认先用本文件判断它属于哪一类：

- 强化核心价值
- 只是改善现有主路径
- 边缘增强
- 明显越过当前产品边界

默认行为应是：

- 优先加深当前核心产品，而不是横向扩展成另一类工具
- 优先补可信度、清晰度和控制力，不优先补花式能力
- 遇到明显跨产品方向的提案时，先停下来确认，而不是直接实现

---

## 13. 什么时候更新本文件

只有在以下情况发生时，才应该更新本文件：

- 产品核心用户发生变化
- 长期主方向发生变化
- 某类原本的非目标已被正式纳入长期路线
- 版本进入新的产品阶段，例如从 `0.x` 进入 `1.x`

如果只是某轮功能执行、一次性修复或短期试验，不应频繁改动本文件。
