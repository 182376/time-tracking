# 应用映射与分类治理方案（执行版）

## 1. 背景与问题

当前 `src/lib/ProcessMapper.ts` 中的 `MAPPINGS` 主要靠手写维护，已经暴露三个核心问题：

- 映射覆盖率低：本机常用应用无法完整命中。
- 分类粒度粗：只有少量大类，无法支撑真实使用分析。
- 颜色体系混乱：颜色由单个映射随意指定，缺少统一规范，视觉不稳定。

这类问题不能继续靠临时补丁逐个修，需要改为“规则中心 + 自动识别 + 渐进治理”。

## 2. 目标

- 应用识别：同一应用多进程形态统一归并（如 `xxx.exe`/`xxx_tray.exe`/`xxx_widget.exe`）。
- 分类准确：常见应用自动落到正确类别，未命中时可稳定降级。
- 颜色一致：应用使用“应用主题色”，专注分布使用“类别色”，规则统一且可解释。
- 类别扩展：类别数量提升到可用粒度，避免“大类过粗导致分析失真”。
- 规则可变：`MAPPINGS` 支持在设置页做用户级覆盖调整，不再只能改源码。
- 可扩展：新增应用不需要提前手写映射也能有合理展示。

## 3. 非目标（本阶段不做）

- 不做云端模型分类（先保证本地可控、可测试、可回滚）。
- 不做一次性“全宇宙应用库”（采用迭代扩展策略）。
- 不让用户承担数据修正责任（用户只看结果，不做手工排障）。

## 4. 架构方案

### 4.1 统一身份层（Identity）

将应用身份从“只看 `exe_name`”升级为“多信号判定”：

- `canonicalExe`：可执行名归一化（含别名归并）。
- `productName`：进程文件元数据（可从 PE 版本信息提取）。
- `companyName`：厂商名（用于提升分类置信度）。
- `processPath`：安装路径特征（如 `Program Files`、`WindowsApps`）。
- `windowTitle`：仅作为弱信号，不直接决定身份。

最终身份键：`appKey = canonicalExe`（保持统计稳定），其余字段用于分类决策和展示优化。

### 4.2 分类引擎（Rule Engine）

分类采用三层优先级，避免纯手工映射：

1. 强规则命中（高置信）
- 明确别名与核心应用规则（如抖音、微信、QQ、浏览器、IDE）。
- 系统进程排除规则。

2. 关键字与元数据推断（中置信）
- 基于 `productName/companyName/processPath` 的关键词规则。
- 例如 `JetBrains / VSCode / Cursor` 优先归为开发类。

3. 降级策略（低置信）
- 未命中应用自动生成展示名并标记为“未分类（other）”。
- 保证可见、可统计，但不污染已知分类规则。

### 4.3 类别体系升级

将当前粗粒度类别升级为稳定枚举（建议 14 类）：

- `development` 开发编码
- `office` 办公协作
- `browser` 浏览阅读
- `communication` 即时通讯
- `meeting` 会议语音
- `video` 视频内容
- `music` 音乐音频
- `game` 游戏
- `design` 设计创作
- `reading` 阅读研究
- `finance` 金融交易
- `utility` 工具效率
- `system` 系统进程（仅内部识别，不进入用户统计）
- `other` 未分类

说明：UI 可按产品策略做上层聚合（例如展示为“工作学习/娱乐/社交”），但底层统计必须保留细粒度类别。

### 4.4 颜色规范

颜色从“应用随意配置”改为“统一双轨模型”：

- 应用主题色：用于应用排行、智能时间轴、应用分布列表条。
- 类别色：仅用于“专注分布”这类类别聚合图。

规则：

1. 应用主题色定义（硬规则）
- 应用主题色 = 应用图标主体颜色（dominant color）。
- 主题色来源只认图标主色，不使用手写品牌色作为常规来源。
- 若当前样本无图标（或图标提取失败），仅使用“临时兜底色”，并在后续图标可用时回填为图标主色。

2. 组件颜色使用规则
- 应用排行：应用主题色
- 智能时间轴：应用主题色
- 应用分布（左侧列表）：应用主题色
- 专注分布（圆环）：类别色

3. 一致性约束
- 同一 `appKey` 在任意页面必须保持同一应用主题色。
- 同一类别在任意日期必须保持同一类别色。

### 4.5 `other` 人工判定入口（设置页）

为降低 `other` 长尾误差，增加一个“低干扰”的设置页组件，专门处理已出现但未自动识别的应用。

组件要求：

- 仅显示“用户真实出现过”的 `other` 应用（来源：本地 sessions）。
- 默认弱视觉样式（折叠态 / 次级分组），不抢主设置区注意力。
- 每条只提供最小决策动作（选择类别），不暴露技术细节。
- 用户改判后，后续统计按新类别生效；原始 session 事实数据不改写。

数据范围与排序：

- 范围：仅显示近一段时间（例如近 30 天）出现过的 `other` 应用。
- 排序：按最近出现时间 + 累计时长降序。
- 去重：按 canonical executable 聚合，避免 `tray/widget` 重复展示。

### 4.6 `MAPPINGS` 可变区治理（设置可改）

将 `const MAPPINGS: Record<string, AppInfo>` 从“源码硬编码单层结构”升级为“默认层 + 用户覆盖层”：

1. 规则分层
- 默认层（内置）：`defaultMappings`，随版本发布，作为稳定基线。
- 覆盖层（用户）：`userOverrides`，由设置页写入本地数据库（按用户设备生效）。
- 运行态：`effectiveMappings = merge(defaultMappings, userOverrides)`。

2. 可编辑字段（首期）
- `category`：允许用户改判类别（最高优先级的业务字段）。
- `displayName`：允许用户修正显示名（可选）。
- `enabled`：是否启用该用户覆盖（支持临时关闭）。

3. 不开放字段（首期）
- `system` 过滤规则（避免误把系统进程放进统计）。
- canonical alias 规则（避免破坏会话归并稳定性）。
- 原始 session 事实字段（保持数据不可变）。

4. 决策优先级（高 -> 低）
- 系统进程排除（硬规则）
- 用户覆盖层（设置页）
- canonical alias 归并
- 默认 `MAPPINGS`
- 自动识别规则（关键词/元数据）
- `other` 降级

5. 回退机制
- 单条应用支持“恢复默认”。
- 全局支持“清空全部用户覆盖”。
- 覆盖变更写审计时间戳，便于回放和排查。

## 5. 数据与配置设计

新增可维护配置文件（建议）：

- `src/lib/config/app-rules.json`：规则表（别名、关键词、分类、展示名优先级）。
- `src/lib/config/category-tokens.ts`：类别枚举与颜色 token。

新增持久化覆盖（建议）：

- `settings` 表新增前缀键：`__app_override::<canonicalExe>`。
- value 建议使用 JSON：`{ "category": "...", "displayName": "...", "enabled": true, "updatedAt": 0 }`。

保留代码入口：

- `processNormalization.ts`：归一化与系统排除。
- `ProcessMapper.ts`：从“硬编码大表”转为“规则执行器 + fallback + userOverrides”。

## 7. 质量指标（持续追踪）

- 应用识别合并率：同应用多 exe 合并成功比例。
- 未分类占比：`other` 时长 / 总时长。
- 分类准确率：抽样人工校验正确比例。
- 应用颜色一致性：同 appKey 颜色稳定比例（目标 100%）。
- 类别颜色一致性：同类别颜色稳定比例（目标 100%）。
- 覆盖稳定性：用户覆盖跨重启一致生效比例（目标 100%）。

## 8. 回滚策略

- 规则配置版本化（保留上一个稳定版本）。
- 新规则灰度开关（可快速切回旧规则）。
- 分类异常时只回滚规则层，不回滚 session 原始数据。

## 9. 当前决议

- 后续不再直接在 `MAPPINGS` 中临时补条目。
- 先落“规则中心化 + 分类体系 + 颜色规范”，再做扩展词库。
- 所有新增分类修复必须先补测试再改实现。
