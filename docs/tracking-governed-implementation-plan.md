# 计时系统正规化实施方案（执行版）

## 1. 目标与范围

本方案用于把当前“问题驱动补丁”升级为“架构驱动实施”，保证以下结果长期稳定：

- 应用时长统计（应用排行、应用分布、每日总时长）只反映真实活跃时长。
- 系统进程不进入用户统计视图。
- 同一应用的变体进程（如 `douyin.exe` / `douyin_tray.exe` / `douyin_widget.exe`）统一归并。
- 计时链路故障时，系统自动降级与恢复，不把排障责任转移给用户。

不在本阶段范围内：

- 手工维护海量应用映射表。
- 通过用户手动操作修正基础计时数据。

## 2. 核心原则

- 分层治理：采集层、标准化层、展示层职责分离。
- 单一事实来源：原始 session 只存事实，不做展示推断。
- 统一口径：所有“统计口径”由标准化层统一输出，UI 只消费。
- 先验收再发布：每个阶段必须满足验收门槛后才能进入下一阶段。
- 禁止临时补丁：新增修复必须先落到本方案对应阶段任务中。

## 3. 架构治理模型

### 3.1 采集层（Rust Tracker）

- 负责：前台窗口采样、AFK 判定、会话开关、故障保护。
- 产出：原始 session 事件（`exe_name`、`process_path`、`window_title`、时间边界）。
- 约束：不承担应用分类与统计归并逻辑。

### 3.2 标准化层（Session Compiler / Read Model）

- 负责：可追踪过滤、别名归并、统计口径统一、时间轴展示归并。
- 产出：
  - 统计模型（真实活跃时长）
  - 时间轴模型（允许展示合并，不反向污染统计）
- 约束：所有业务页面必须通过该层读取，不可绕过。

### 3.3 展示层（React UI）

- 负责：展示标准化层输出，不新增本地统计推断。
- 约束：不显示内部诊断实现细节，不向用户暴露排障噪声。

## 5. 变更流程（强制）

所有后续修复必须遵循：

1. 问题归类到阶段任务，不允许直接临时修补。
2. 先补测试（或先写失败用例），再改实现。
3. 本地验证通过后再进入提交。
4. 更新本文件勾选状态与阶段备注。

## 6. 当前执行决议

- 本项目后续计时相关改动，统一按本方案执行。
- 若出现线上问题，先定位到阶段任务，再进入实现，不再直接点改。
