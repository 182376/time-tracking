# Quiet Pro 组件规范

## 1. 文档定位

本文件定义本项目长期使用的 `Quiet Pro` UI 规范。

它不是某一轮美化任务的执行单，也不是某个页面的局部设计说明，而是以后新增组件、扩展页面、微调交互时都应遵循的长期基线。

如果某次任务没有明确给出新的设计方向，就默认继续遵循本文件。

---

## 2. 与其他长期文档的关系

本文件主要回答：

- 这个产品的长期 UI 气质是什么
- 新组件应该长成什么样
- 什么样的“设计感”是允许的
- 什么样的“高级感”其实是在越界

它与其他长期文档的关系如下：

- 与 [`architecture-target.md`](./architecture-target.md) 互补，`architecture-target` 约束代码边界，本文件约束视觉与组件边界。
- 与 [`issue-fix-boundary-guardrails.md`](./issue-fix-boundary-guardrails.md) 互补，边界守则决定“能不能直接修”，本文件决定“允许怎么设计”。
- 与 [`time-tracker-app-icon-design-brief.md`](./time-tracker-app-icon-design-brief.md) 互补，图标 brief 是产品标识专项 brief，本文件是所有 UI 的长期母规则。

如果某个局部 UI 方案与本文件冲突，应优先确认是否真的需要新的长期设计方向，而不是默认局部方案优先。

---

## 3. Quiet Pro 一句话定义

`Quiet Pro = 安静、专业、克制、可长时间使用的桌面效率工具界面。`

它不追求“第一眼惊艳”，而追求：

- 信息清楚
- 层级稳定
- 长时间使用不累
- 不像 AI 模板
- 后续容易继续扩展

---

## 4. 当前产品语境

本项目是一个：

- Tauri 桌面应用
- 时间追踪与工作陪伴型工具
- 以 `Dashboard / History / Classification / Settings` 为核心工作面板的产品

因此，UI 的默认气质应更接近：

- 桌面工具
- 工作台
- 稳定系统
- 长时间持续使用的软件

而不是更接近：

- 营销站
- 创意展示页
- 娱乐型效率工具
- 视觉实验品

Quiet Pro 的价值不在“显得很会设计”，而在“显得可信、耐用、有秩序”。

---

## 5. 核心设计原则

### 5.1 信息优先于装饰

视觉的第一职责是帮助用户理解信息和完成操作，不是制造“设计感”。

优先使用这些手段建立质量感：

- 字重
- 间距
- 对齐
- 留白
- 信息分组
- 语义清楚的状态层级

### 5.2 中性系统优先于单页特效

界面整体应保持中性、专业、耐用。

即使页面有重点区域，也应该通过系统化样式表达，而不是靠单页独有的特殊效果抢眼。

### 5.3 产品气质重于内容皮肤

本项目更像桌面效率工具，而不是写作主题或营销页面。

因此组件应优先体现：

- 可靠
- 清晰
- 稳定
- 可扫读

而不是“个性化装饰”。

### 5.4 主题化建立在骨架稳定之上

允许后续继续换主题，但主题变化应基于稳定骨架：

- token
- 语义颜色
- 圆角尺度
- 边框/阴影尺度
- 组件密度

不应依靠页面里到处散落的硬编码样式来“换皮”。

### 5.5 好看必须建立在可复用之上

Quiet Pro 不反对设计感，但要求这种设计感可以被系统化复用。

如果某个效果：

- 只能在一个页面成立
- 依赖单页独有背景
- 依赖大量硬编码值
- 复制到其他页面就违和

那它通常不属于 Quiet Pro 认可的长期做法。

---

## 6. 允许的整体气质与明确禁止

### 6.1 允许的整体气质

- 冷静
- 克制
- 专业
- 中性
- 桌面工具感

### 6.2 明确禁止

- 玻璃拟态
- 重模糊
- 霓虹发光
- 大面积炫目渐变背景
- 过度圆角
- 为了“高级感”堆大阴影
- 单页面专属花哨特效
- 明显 AI 生成模板感

### 6.3 对“高级感”的正确理解

Quiet Pro 的高级感来自：

- 排版干净
- 层级明确
- 控件一致
- 颜色克制
- 节奏稳定
- 比例准确

不是来自：

- 大面积装饰
- 发光
- 强品牌色占满页面
- 到处都有浮层感
- 为了显贵而故意做复杂

---

## 7. Token 与样式规则

### 7.1 一切优先走语义 token

新增视觉角色时，优先：

1. 复用现有 token
2. 若现有 token 语义不够，再新增语义 token

不要直接在组件里硬编码：

- 新颜色
- 新圆角
- 新阴影
- 新边框风格
- 新动效尺度

### 7.2 Token 的角色应该是语义化的

例如应优先存在这类语义：

- `panel`
- `control`
- `chip`
- `status`
- `track`
- `muted text`
- `danger`

而不是只新增“看起来像这个页面要用的某个值”。

### 7.3 分类色与状态色的边界

分类色、图表色、状态点颜色允许相对丰富，但它们只服务于业务识别。

周围界面 chrome 仍应保持 Quiet Pro 的中性骨架，不要让彩色元素反过来主导整个页面。

### 7.4 样式尺度应当是可预测的

长期上，控件应尽量收敛在有限的尺度档位：

- 有限的圆角档位
- 有限的 padding 档位
- 有限的字级层级
- 有限的边框和阴影强度

Quiet Pro 不鼓励“这个页面再发明一个更顺眼的新档位”。

---

## 8. 页面骨架与原型

### 8.1 壳层

应用壳层应保持：

- 稳定导航
- 清晰页头
- 主内容区边界明确
- 页面切换不过分戏剧化

壳层应服务于长期使用秩序，而不是承担视觉表演。

### 8.2 面板

面板是本项目最常见的容器原型。

面板应体现：

- 清晰分组
- 克制边界
- 足够留白
- 不靠重阴影建立层次

如果一个页面里有多个面板，只能有一个主要视觉焦点，其余应主动降噪。

### 8.3 页头

页头应承担：

- 页面身份识别
- 简短上下文提示
- 右侧必要 controls / status / actions

页头不应承担：

- 大片品牌展示
- hero 式视觉主舞台
- 页面私有的强视觉实验

### 8.4 列表与工作台

列表、映射表单、设置区块应更像“工作台”，而不是卡片墙。

因此要优先保证：

- 行级可扫读
- 信息密度平衡
- 操作入口稳定
- 状态对比明确

---

## 9. 组件规则

### 9.1 组件原型

新组件应优先归入已有原型之一：

- `panel`
- `control`
- `chip`
- `status`

若无法归类，再考虑新增原型，但需要非常谨慎。

### 9.2 状态完整性

新组件至少应定义：

- `default`
- `hover`
- `active`
- `focus`
- `disabled`

需要时再补：

- `loading`
- `empty`
- `selected`
- `error`

不能只做静态样子，不定义交互状态。

### 9.3 图标规则

图标是辅助识别，不是主角。

因此：

- 图标尺寸应克制
- 图标颜色不应抢走主要信息层级
- 图标背景应服务于识别，不应变成装饰噱头

如果某个设计方案的成立前提是“图标必须特别抢眼”，那通常已经偏离 Quiet Pro。

### 9.4 按钮与输入控件

按钮和输入控件应表现为：

- 利落
- 清晰
- 可预测

不要：

- 过分圆润
- 过度拟物
- 通过强烈阴影制造“高级”

### 9.5 状态组件

badge、chip、status 一类组件应服务于：

- 当前状态
- 风险提示
- 筛选条件
- 轻量标签

不应被滥用为：

- 大块装饰
- 标题替代物
- 没有语义必要的“彩色点缀”

---

## 10. 排版、密度与数据展示

### 10.1 排版

Quiet Pro 的视觉质量很大一部分来自排版。

要点：

- 标题层级清楚
- 副文案简短
- 数字与时间信息可快速扫读
- 长行说明尽量少

涉及时间、时长、百分比、计数时，优先考虑稳定的数字对齐体验。

### 10.2 密度

本项目是效率工具，不是海报。

默认应保持：

- 比营销站更高的信息密度
- 比传统后台更克制的噪音
- 在紧凑和可读之间取中间偏专业的平衡

### 10.3 图表与分类色

图表允许使用分类色，但需要遵守两条：

- 分类色要能分辨，不要大面积撞成同一色族
- 图表周边 UI 不要跟着一起变得花

如果分类色较丰富，应主动压低：

- 轨道色
- 面板边框
- 背景层次

这样业务颜色才不会把整个界面拉乱。

---

## 11. 动效规则

Quiet Pro 的动效应当：

- 短
- 轻
- 有反馈意义
- 不打断使用节奏

可以有：

- 页面切换的轻过渡
- 悬停反馈
- toast / modal 的短入场

不应有：

- 大幅位移动画
- 浮夸弹跳
- 为了“灵动”而增加存在感的表演性动画

去掉动效后，组件和页面仍应成立。

---

## 12. 实现偏好

- 先扩展设计系统，再做页面局部 workaround
- 能复用现有模式，就不要为单个场景发明新视觉语言
- 无明确需求时，不改变现有功能语义和操作路径
- 若某个提议明显偏离 Quiet Pro，先停下确认，不要自行发散

Quiet Pro 的默认实现倾向永远是：

- 更克制
- 更稳定
- 更少一次性特例
- 更易长期维护

---

## 13. 新增组件前检查

提交前至少自查这几件事：

- 这个组件是否复用了现有 token
- 是否引入了新的视觉档位
- 是否比页面主信息更抢眼
- 状态是否完整
- 放进 `Dashboard / History / Classification / Settings` 里是否违和
- 去掉动效后，这个组件是否仍然成立
- 它是否可以被其他页面复用，而不是只能活在当前页面

---

## 14. 何时必须先确认

出现以下情况时，不要直接做：

- 需要新增明显不同的视觉语言
- 需要引入强品牌化方向
- 需要改变页面密度策略
- 需要使用大面积彩色背景或新的装饰性手法
- 需要改变 Quiet Pro 的中性骨架
- 需要让某个单页显著偏离 `Dashboard / History / Classification / Settings` 的长期气质

这类情况必须先和用户确认方向。

---

## 15. 给 Codex 的默认执行方式

- 默认把 Quiet Pro 当作唯一 UI 基线
- 新增组件时优先复用现有模式与 token
- 如果某项设计决策没有明示，选择更克制、更稳定、更易长期维护的方案
- 若某个实现只是“看起来更炫”，但会削弱可读性或一致性，应拒绝那种做法
- 若某个方案只能通过页面私有特效成立，应视为长期风险而不是设计亮点
