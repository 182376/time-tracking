# 计时算法优化方案

## 文档用途

这份文档专门记录 `tracking / timing` 算法层面的后续优化方向。

它和总路线图的区别是：

- 总路线图关注整体架构职责边界
- 这份文档只关注“计时为什么准、为什么会错、下一步怎么收敛”

## 本次故障样本

以 `2026-04-04` 这次真实故障为例，出现了两个非常典型的问题：

1. 用户切到抖音后，界面顶部仍显示 `正在追踪: VSCodium`
2. 用户在 `11:30` 左右离开去吃饭，历史里仍然持续记录了 `VSCodium`

复盘数据库后，关键信息如下：

- 最后一条未结束 session 是 `VSCodium.exe`
- 该 session 开始于 `2026-04-04 11:22:26`
- tracker heartbeat 最后一次更新时间是 `2026-04-04 11:26:32`
- 之后 tracking runtime 没有继续推进，导致旧 session 一直挂着

这说明问题不只是“UI 显示不准”，而是“底层计时管线失活后没有及时自恢复，也没有及时把旧 session 封口”。

## 目标

- 前台窗口切换后，计时归属应尽快切到正确应用
- 用户离开、锁屏、睡眠、崩溃后，旧 session 应尽快被可靠封口
- tracker 失活时，不应继续把旧应用当作“当前活跃”
- 历史页和概览页要能区分“真实活跃”与“残留脏数据”
- 优化要分阶段推进，优先做高收益、低风险项

## 当前主要问题

### P1. 采样链路缺少超时与健康保护

目前前台窗口采样依赖 Windows API。
如果采样链路卡住，tracking runtime 就可能停在旧状态，之后所有计时都会持续挂在最后一个应用上。

### P2. runtime 失活后缺少“状态降级”

现在前端一旦拿到最后一个 `active-window-changed`，就会一直显示那个应用。
如果后端 tracker 停掉了，前端不会自动把状态切为“未知 / 已中断”。

### P3. 离开检测仍主要依赖 AFK 输入空闲

当前 AFK 主要靠 `GetLastInputInfo`。
这对“锁屏 / 系统睡眠 / 恢复 / 前台窗口瞬时不可见”这些场景还不够稳。

### P4. session identity 仍然偏粗

当前 session 切换主判定主要看 `exe_name`。
这有利于减少编辑器里频繁切标签造成的碎片，但也意味着：

- 同一进程内的不同顶层窗口无法做更细粒度判断
- 诊断时缺少 `hwnd / process_id / class_name` 这类上下文

### P5. 展示层缺少“数据可信度”信号

当前历史页和概览页会忠实展示库里的 session。
如果底层 session 因故障没有结束，展示层没有能力告诉用户“这段记录可能不可信”。

## 优化原则

- 先保证“不会长时间挂错应用”
- 再保证“离开后能可靠截断”
- 再提升“窗口识别质量”
- 除智能时间轴外，其余应用时长、排行、小时分布、7 天汇总都只基于真实 active duration
- 智能时间轴的合并只服务于“用户更容易看懂时间段主要在做什么”，不能反向污染统计时长
- 最后再做更细的 app 级 heuristic

## 分阶段计划

### 阶段 A：运行时保活与自恢复

目标：
避免 tracker 卡住后把旧应用一直当作当前应用。

任务清单：

- [x] tracking runtime 启动失败后自动重试，而不是一次失败就永久停掉
- [x] 等待 SQLite pool 时持续重试，并输出更明确日志
- [x] 前台窗口采样增加超时保护，避免单次采样卡死拖垮整条计时链路
- [x] 为 tracker 增加“最后成功采样时间”状态
- [x] 前端在 tracker 长时间无心跳时，不再继续显示旧应用为“当前活跃”

验收标准：

- [x] runtime 启动时不会因为初始化时序问题直接永久失活
- [x] 单次采样异常不再轻易导致整段 session 挂死
- [x] tracker 失活时，界面不会继续长期误导显示旧应用

### 阶段 B：离开边界纠正

目标：
把“去吃饭 / 锁屏 / 睡眠 / 崩溃恢复”这些边界做稳。

任务清单：

- [x] 补锁屏 / 解锁事件处理
- [x] 补睡眠 / 唤醒事件处理
- [x] 为“超长未结束 active session”增加守护封口策略
- [x] 明确 heartbeat 与实际成功采样之间的关系，避免异常情况下高估活跃时长

验收标准：

- [x] 离开设备后不会长时间继续累计旧应用时长
- [x] 系统恢复后旧 session 会被截断在合理边界

### 阶段 C：窗口身份质量提升

目标：
让“当前到底是谁在前台”更可解释、更可诊断。

任务清单：

- [x] 在采样模型中补 `hwnd`
- [x] 在采样模型中补 `process_id`
- [x] 评估并补充 `window class` / `root owner` 信息
- [x] 为 session 切换逻辑引入更明确的 identity 结构，而不只是 `exe_name`

验收标准：

- [x] 事件 `reason` 能解释“为什么这次没有切 session”
- [x] 同 exe 多顶层窗口的行为更可控

### 阶段 D：展示层可信度与自动诊断

目标：
当底层数据可疑时，由软件自动识别、自动截断、自动标记，而不是把排查责任交给用户。

任务清单：

- [x] 自动识别 tracker stale 场景下的“异常未封口 session”
- [x] 在历史页或概览页自动标记“疑似残留 / 待确认”的记录
- [x] 增加简洁的 tracker 健康诊断入口
- [x] 明确约束：应用排行、应用分布、小时分布、7 天汇总都只统计真实 active duration，不受时间轴展示合并影响

验收标准：

- [x] 用户能分辨“真实活跃”与“可疑残留记录”
- [x] 故障排查不必第一时间手查 SQLite
- [x] 除智能时间轴外，其余时长统计不会因为展示合并而变大

## 当前已落地项

本轮已经先落地了这些高优先级项：

- runtime 启动失败自动重试
- SQLite pool 等待改为持续重试
- 前台窗口采样增加超时保护
- 记录“最后成功采样时间”，和 heartbeat 分开使用
- 前端基于最后成功采样时间做健康检查，超时后不再显示旧应用为“当前活跃”
- tracker watchdog 会在主循环卡住后主动封口过长未结束 session
- 主循环恢复后，如果发现 active session 已被 watchdog 封口，会自动补开新 session
- Windows 锁屏 / 解锁事件会直接进入 Rust 侧计时链路
- Windows 睡眠 / 唤醒事件会在系统边界处封口旧 session
- 采样模型现在会额外记录 `hwnd` 和 `process_id`
- 采样模型补充了 `root_owner_hwnd` 与 `window_class`，同 exe 多窗口识别更稳定
- session 切换不再只看 `exe_name`，而是基于 `app_key + process_id + root_owner + window_class` 的 identity
- transition 事件通过 `reason` 明确区分“应用切换 / 状态切换 / 元数据刷新”
- `LockApp.exe` / `LogonUI.exe` 已加入系统级排除列表，避免把锁屏界面误算进用户时长
- tracker stale 时，前端不再继续让 live session 本地增长，而是截断到最后一次成功采样
- Dashboard / History 会自动汇总并标记待确认记录，不要求用户手工查库
- 应用排行、应用分布、小时分布、7 天统计继续基于真实 active duration；智能时间轴仅负责展示合并

这几项主要解决“tracking runtime 一旦失活，整段时间全挂在旧应用上”的问题。

## 后续可选优化顺序

1. 根据真实样本继续细化同 exe 多窗口识别策略（例如浏览器主窗与弹窗的业务归并规则）
2. 在不增加用户操作负担的前提下，补充更轻量的诊断信息出口（如最近一次 transition reason 快照）
3. 持续用回归测试覆盖新增 heuristic，避免误伤现有计时准确性

## 完成定义

当以下条件成立时，可以认为计时算法这一轮基本稳定：

- [x] tracker 失活不会再长时间挂错旧应用
- [x] 离开设备后 session 能在合理边界结束
- [x] 前台窗口识别结果更可解释、可诊断
- [x] UI 能在数据异常时给出明确提示
