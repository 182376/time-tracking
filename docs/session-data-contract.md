# Session 数据契约与迁移规范

## 1. 目标

本文定义计时系统的统一数据契约，避免“原始数据、统计数据、展示数据”混用导致口径漂移。

## 2. 原始 Session（事实层）

表：`sessions`

- `id`: 自增主键，仅用于唯一标识记录。
- `app_name`: 采集时写入的展示名候选值，可变，不作为统计主键。
- `exe_name`: 采集时进程可执行名（原始值），不保证稳定，不直接用于最终统计分组。
- `window_title`: 采集时窗口标题（原始值），仅用于时间轴辅助展示。
- `start_time`: session 开始时间（ms 时间戳）。
- `end_time`: session 结束时间（ms 时间戳，可空表示进行中）。
- `duration`: 记录时长（ms，可空；读模型会二次修正）。

约束：

- 原始层不做业务归并，不做系统进程语义判断。
- 原始层字段可保留噪声，标准化层负责清洗与统一。

## 3. 标准化输出（读模型层）

模块：`src/lib/services/sessionCompiler.ts`

- `appKey`: 规范化可执行键（canonical executable），是统计与合并的唯一主键。
- `displayName`: 用户可见应用名，优先使用 canonical 规则，其次使用 `app_name`。
- `statsExeName`: 最终用于统计项标识的 `exe_name` 输出值。
  - 若原始 `exe_name` 已与 `appKey` 一致，保留原值。
  - 否则输出 `appKey`，保证同应用变体统一归并。

规则来源：

- 统一在 `src/lib/processNormalization.ts` 定义：
  - 别名归并（如 `douyin_tray` -> `douyin.exe`）
  - 系统进程过滤（含 `PickerHost`）
  - canonical 展示名（如 `douyin.exe` -> `抖音`）

## 4. 统计口径约束

- 应用排行、应用分布、总时长、小时分布：只基于真实 active duration 计算。
- 时间轴合并：仅用于展示连续行为，不反向改变统计口径。
- UI 只消费读模型，不自行推导统计结果。

## 5. 历史数据处理策略

当前策略：**只读过滤 + 读时归并**（不立即重写历史库）

- 优点：无迁移风险、可即刻生效、回滚成本低。
- 风险：旧脏数据仍在库内，但不会进入统计与展示结果。

后续可选策略：离线迁移脚本

- 把历史别名统一改写为 canonical key。
- 把系统进程历史记录软删除或迁移到归档表。

## 6. 回滚策略

代码回滚：

- 回滚 `processNormalization.ts` 与 `sessionCompiler.ts` 相关提交即可恢复旧逻辑。

数据回滚：

- 因当前采用只读过滤策略，不涉及强制数据改写，无需数据回滚。
- 若未来启用离线迁移，必须先做 DB 备份并记录迁移版本号。

## 7. 验收清单

- [ ] 同应用变体不会在应用排行中分裂多条。
- [ ] 系统进程不会出现在应用排行与分布中。
- [ ] 时间轴展示策略变化不会影响统计总时长。
- [ ] `npm test` / `npm run build` / `cargo test` 全部通过。
