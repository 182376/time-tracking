## 1. 背景

当前应用顶部的最小化、最大化、关闭按钮，仍然来自操作系统原生窗口栏，而不是 Quiet Pro 自有 UI。

本轮问题不是“它能不能工作”，而是：

- 它是否已经偏离 Quiet Pro 的统一性目标
- 如果改成自定义标题栏，收益是否足以覆盖工程和维护成本
- 如果真的要做，正确的 Quiet Pro 方向应该是什么

因此当前状态应理解为：

`系统原生窗口壳 + Quiet Pro 产品内容区`

这不是错误实现，只是一个尚未做“最上层窗口壳统一”的设计选择。

## 3. 需要回答的核心问题

是否值得把当前系统原生窗口栏，改成 Quiet Pro 自定义标题栏。

这个问题本质上不是单纯的视觉选择，而是以下两类收益与成本的取舍：

- 视觉统一性收益
- 窗口行为、工程复杂度、长期维护成本

## 4. 两条可选路线

### 4.1 路线 A：继续保留系统原生窗口栏

含义：

- 顶部最小化 / 最大化 / 关闭继续由 Windows 提供
- Quiet Pro 的责任边界从应用内容区开始

优点：

- 最稳
- 最符合桌面应用平台习惯
- 自动保留拖拽、吸附、双击标题栏最大化、系统菜单等原生行为
- 不需要为窗口层维护额外前端和 Tauri 逻辑
- 不容易把桌面产品做成“网页导航栏套壳”

缺点：

- 窗口最上层与 Quiet Pro 内容区存在一层视觉切换
- 整体产品统一感弱于完全自定义标题栏

适用情况：

- 你更重视稳定、平台一致性和低维护成本
- 你认为当前最上层系统栏虽然不是 Quiet Pro，但也不至于构成明显体验问题

### 4.2 路线 B：改成 Quiet Pro 自定义标题栏

含义：

- 去掉系统原生窗口装饰
- 在应用壳层内实现一条自定义 Quiet Pro 标题栏
- 自己接管最小化 / 最大化 / 关闭与拖拽等行为

优点：

- 从窗口最上层开始统一为 Quiet Pro
- 整个产品从顶到底更完整、更一体
- 可以把标题、应用图标、少量全局状态放到同一条克制的壳层里

缺点：

- 工程复杂度明显上升
- 需要自己接窗口控制行为
- 拖拽区、点击区、双击标题栏行为、最大化后的边缘表现都要自己兜住
- 以后每次改顶部布局，都会带出窗口行为回归风险
- 稍微做重一点，就会比现在更不像 Quiet Pro

适用情况：

- 你明确认为“最上层系统窗口壳已经影响产品完整感”
- 你愿意承担一层长期维护成本，换取更高的一体化

## 5. 自定义标题栏的真实后果

如果选择路线 B，后果主要在这 5 个方面。

### 5.1 窗口行为接管

原本由系统免费提供的行为，需要应用自己接手：

- 最小化
- 最大化 / 还原
- 关闭
- 拖拽移动
- 双击标题栏最大化 / 还原

如果这些行为没处理完整，用户会第一时间觉得“不像桌面应用”。

### 5.2 平台原生体验变脆

原生窗口栏天然带有平台习惯。

一旦自定义，以下问题都可能出现：

- 拖拽区和按钮区冲突
- 最大化后顶边 / 圆角 / 留白不自然
- 吸附和窗口边缘观感不协调
- 关闭按钮 hover 风险提示不够清楚

这类问题不是功能 bug，但会持续消耗产品质感。

### 5.3 维护成本长期存在

自定义标题栏不是“一次做完就永久结束”的活。

后续任何顶部相关改动，都可能波及：

- 壳层布局
- 拖拽区
- 窗口控制按钮
- 最大化状态样式
- 平台行为回归

### 5.4 Quiet Pro 风险

Quiet Pro 的目标是安静、克制、桌面工具感。

标题栏如果处理不好，很容易出现这些反方向结果：

- 顶栏太厚
- 按钮太像网页 navbar controls
- 图标和状态堆得太满
- 顶部变成“产品头图”而不是安静壳层

这会直接破坏 Quiet Pro。

### 5.5 架构影响

从架构角度，这轮改动不会触发业务层大改，但会引入一条新的壳层职责链：

- Tauri 窗口配置
- 前端 AppShell 顶部标题栏组件
- 窗口控制调用边界

因此它虽然不是大重构，但也绝不是单纯换样式。

## 6. 如果要做，正确的 Quiet Pro 设计方向

如果最终决定做 Quiet Pro 自定义标题栏，推荐方向必须非常克制。

### 6.1 标题栏骨架

- 高度保持薄，建议约 `36-40px`
- 左侧只放应用图标和标题
- 中间尽量保留为拖拽区
- 右侧只放最小化 / 最大化 / 关闭

### 6.2 视觉原则

- 背景与应用壳层同体系
- 用轻边框或轻分割表达层级
- 不做厚重阴影
- 不做玻璃、发光、强渐变
- 不把顶部做成品牌型导航栏

### 6.3 按钮原则

- 最小化 / 最大化：低存在感控制按钮
- 关闭：允许更明确的危险态，但仍保持 Quiet Pro 克制
- 图标应服务识别，不抢视觉中心

### 6.4 明确禁止

- 顶部塞入完整页面导航
- 把标题栏做成 marketing hero
- 堆很多状态标签、搜索框、筛选器
- 做成明显网页感 header

## 7. 推荐结论

从当前项目状态看，我的默认建议是：

### 7.1 默认推荐

优先继续保留系统原生窗口栏。

原因：

- 当前内容区已经基本完成 Quiet Pro 收口
- 最上层窗口栏虽然不是 Quiet Pro，但仍然是合理的桌面产品实现
- 自定义标题栏的主要收益是“统一感更强”，不是“解决严重产品问题”
- 它带来的维护成本是真实长期成本

### 7.2 何时值得改

只有在以下判断成立时，才值得正式启动自定义标题栏任务：

- 你明确认为最上层系统窗口栏已经持续破坏产品完整感
- 你愿意接受这是一层长期维护点
- 你接受这轮任务会同时涉及前端壳层与 Tauri 窗口控制边界

## 8. 决策门槛

在真正进入执行前，建议按下面标准做最终决定。

### 8.1 适合继续保留原生窗口栏

如果你更看重：

- 稳定
- 平台原生体验
- 低维护成本
- 不额外引入窗口层复杂度

那就不做。

### 8.2 适合启动自定义标题栏

如果你更看重：

- 整个产品从顶到底都统一成 Quiet Pro
- 顶层系统窗口壳已经明显影响你对产品完成度的判断
- 你愿意多承担一层壳层维护

那就可以做。

## 9. 如果决定正式执行，推荐任务边界

如果后续决定真的进入开发，执行单应严格限制在以下范围：

- Tauri 主窗口装饰配置
- `AppShell` 自定义 Quiet Pro 标题栏
- 最小化 / 最大化 / 关闭按钮接线
- 拖拽区与按钮区分离
- 最大化 / 还原状态下的 Quiet Pro 样式收口

明确不应顺手扩展：

- 不重做整页导航
- 不借机改页面布局
- 不往标题栏塞新的业务入口
- 不扩展到 Rust 业务逻辑

## 10. 本轮决策结论

本轮结论已经确定为：

- 当前顶部窗口控制按钮确实是系统原生，不是 Quiet Pro 自绘
- 这不是错误实现
- 本轮不启动 Quiet Pro 自定义标题栏开发
